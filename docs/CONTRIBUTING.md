# 프로젝트 기여 및 버전 관리 가이드 (Contribution Guide)

이 문서는 프로젝트의 코드 품질을 유지하고, 변경 사항을 체계적으로 관리하기 위한 가이드라인입니다.

## 1. 커밋 메시지 컨벤션 (Commit Message Convention)

우리는 **Conventional Commits** 규칙을 따릅니다. 커밋 메시지는 다음과 같은 구조로 작성합니다.

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 1.1. Type (필수)
작업의 성격을 나타내는 키워드입니다.

| Type | 설명 | 예시 |
| :--- | :--- | :--- |
| **feat** | 새로운 기능 추가 | `feat: 사용자 로그인 기능 추가` |
| **fix** | 버그 수정 | `fix: 홈 화면 업체명 미표시 오류 수정` |
| **refactor** | 코드 리팩토링 (기능 변경 없음) | `refactor: 캘린더 컴포넌트 분리` |
| **docs** | 문서 수정 | `docs: 기여 가이드 문서 추가` |
| **style** | 코드 포맷팅, 세미콜론 누락 등 (로직 변경 없음) | `style: 코드 포맷팅 적용` |
| **test** | 테스트 코드 추가/수정 | `test: 로그인 API 테스트 추가` |
| **chore** | 빌드 태스크, 패키지 매니저 설정 등 | `chore: 패키지 의존성 업데이트` |

### 1.2. Scope (선택)
변경 사항이 영향을 미치는 범위를 명시합니다.
*   예: `auth`, `calendar`, `home`, `api`

### 1.3. Subject (필수)
변경 사항에 대한 간결한 설명입니다.
*   명령문으로 작성 (예: "Fix bug" 대신 "Fixed bug" 지양, 한글의 경우 "~수정", "~추가" 등으로 통일)
*   끝에 마침표(.)를 찍지 않음

### 1.4. Body (선택)
변경 사항에 대한 자세한 설명을 작성합니다. "무엇을", "왜" 변경했는지에 집중합니다.

### 1.5. Footer (선택)
Breaking Change나 이슈 번호(Closes #123)를 명시합니다.

## 2. 버전 관리 전략 (Versioning)

**Semantic Versioning (SemVer)** 방식을 따릅니다.
버전 형식: `MAJOR.MINOR.PATCH` (예: `1.0.0`)

*   **MAJOR**: 하위 호환되지 않는 API 변경이 있을 때 (대규모 개편)
*   **MINOR**: 하위 호환되는 새로운 기능이 추가되었을 때
*   **PATCH**: 하위 호환되는 버그 수정이 있을 때

## 3. 작업 보고서 (Commit Report)

작업 단위(Task)가 완료될 때마다 아래 양식에 맞춰 보고서를 작성하여 이력을 남깁니다.

### 📝 작업 보고서 템플릿

```markdown
## 📅 날짜: 202X-XX-XX
## 🏷️ 작업 종류: [기능추가 / 버그수정 / 리팩토링 / 문서 / 기타]
## 🔢 버전: vX.X.X (이전 버전: vX.X.X)

### 1. 🎯 작업 목표
- (작업의 주된 목적을 한 줄로 요약)

### 2. 🛠️ 주요 변경 사항
- **[컴포넌트/모듈명]**: (변경 내용)
- **[API]**: (변경 내용)
- (구체적인 파일명이나 로직 변경 사항 기술)

### 3. 🐛 버그 수정 (해당 시)
- **증상**: (어떤 문제가 있었는지)
- **원인**: (원인이 무엇이었는지)
- **해결**: (어떻게 해결했는지)

### 4. 📸 스크린샷 / 결과물 (선택)
- (UI 변경이 있는 경우 스크린샷 첨부)
- (API 응답 예시 등)

### 5. ⚠️ 특이사항 / 추후 과제
- (작업 중 발견한 이슈나 나중에 처리해야 할 사항)
```

## 4. 예시 (Example)

### 커밋 메시지 예시
```
fix(auth): 직원 회원가입 시 업체명 누락 문제 수정

관리자(Boss) 데이터에 상호명이 있어도 직원 가입 시 상속되지 않는 문제 해결.
- BossSignupForm: businessInfo에 s_nm 필드 명시적 추가
- auth.js: 회원가입 시 companyName 자동 보정 로직 추가
```

### 작업 보고서 예시
```markdown
## 📅 날짜: 2025-11-28
## 🏷️ 작업 종류: 버그수정
## 🔢 버전: v0.1.2

### 1. 🎯 작업 목표
- 홈 화면에서 업체명과 소속이 '소속 없음'으로 뜨는 치명적 버그 수정

### 2. 🛠️ 주요 변경 사항
- **BossSignupForm.tsx**: `businessInfo` 객체 생성 시 `s_nm` (상호명) 필드가 누락되어 있던 것을 추가함.
- **server/routes/auth.js**: 회원가입 API에서 `companyName`이 비어있을 경우 `businessInfo`에서 추출하여 저장하도록 폴백 로직 구현.

### 3. 🐛 버그 수정
- **증상**: 직원으로 가입 후 홈 화면 진입 시 업체명이 뜨지 않음.
- **원인**: 프론트엔드에서 상호명을 백엔드로 제대로 전달하지 않았고, 백엔드에서도 방어 로직이 없었음.
- **해결**: 데이터 전송 및 저장 로직 양쪽 모두 수정하여 이중으로 안전장치 마련.

### 5. ⚠️ 특이사항
- 기존 가입된 계정은 데이터가 없으므로 재가입 필요함.
```
